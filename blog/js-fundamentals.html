JavaScript and its relationship to HTML and CSS

I want you to imagine two things: a dead body and a rock concert

Now let’s talk about HTML.

In literal terms, on a web page, there is content—words, images, and stuff you see. This is HTML. This content looks pretty dry all on its own, downright horrible, even. So, let’s call HTML a dead body, or perhaps your favorite band playing a great setlist.

Now, onto CSS.

CSS deals with that dry, crusty HTML by adding colors, fonts, and other good stuff —that’s the styling. Pretty much anything you can imagine about how the page should look is handled by CSS. CSS is like when a funeral home does the hair and makeup of the deceased. It looks a lot nicer, but still not a lot of function. Or, we can consider that CSS is like the outfits, lighting, AV, and choice of venue for a rock concert. The content is the same, but now it’s dressed up and ready to impress.

However, the webpage is still the same every time; the dead body doesn’t do anything and the concert is not unique. It’s the same with or without the audience.

JavaScript changes all of this.

JavaScript is like turning that dead body into a marionette puppet, like the lead singer of the band crowd surfing. It’s designing a web page that doesn’t just look the way the creator wants but any way the user wants.

JavaScript powers functionality, customization, personalization, and interactivity by being able to dynamically modify the HTML and CSS.


Control Flow and Loops

In programming, control flow is like the order of steps that tell a program what to do next.

However control flow can be a lot more powerful and complex than just an ordered list of steps.

Every morning I start by drinking coffee.

My first step is to determine if anyone else has made coffee yet, or if I need to make the first brew. This is an IF conditional, which is like a fork in the road or a switch on a railway track. This makes the code to operate in more than one possible way depending on other variable elements.

In the coffee scenario, IF someone has already made coffee, then I don’t need to go any further; I can simply drink the existing coffee. But if no one has made the coffee, then I will need to make that darn coffee.

In JavaScript, that might be represented like this: 

          
            if (coffeeMade) {
              return "Coffee is ready! Time to drink coffee.";
          } else {
              return "No coffee yet. Time to make some coffee.";
          }

In the event that the coffee needs to be made, the first thing I will need to do is grind the beans. So, after pouring my beans into the grinder, I need to turn the grinder crank until the beans are all ground up. But unless I recognize that the beans have been all ground up, I might accidentally turn the grinder crank until the heat death of the universe, which would not be an effective use of my time, and critically, I’ll never actually get to drink my coffee.

So in code, this is where a LOOP is useful.

A loop lets you cycle through an action either a set or dynamic number of times before continuing with the rest of the code. For the coffee, I want to turn the crank until all the coffee is ground. want the DOM is and how you might interact with it?

The DOM is like a tree map detailing everything on a webpage and how each item relates to the others.

The difference between arrays and objects.

Objects and Arrays are two different ways to structure and store data.

An array is an ordered list. It might look like this:

[1, 2, 3].

The defining use behind an array is that each number is in a specific position. The 1 comes first, the 2 is next, and the 3 after that. An array can be made up of anything (even objects); the critical point is that items in an array stay in order and can be counted upon to be in order.

An object structures data in a different way. This is best thought of as a Dictionary.
In a dictionary, each word has a definition. In an Object, each Key has a Value (although the value can be empty).
So in the dictionary, the word is the key, and the value is the definition.

Objects are great for storing sets of similar data.
A personal contact book is a great example of an Object. Each contact is like an object. Each contact has a name and a phone number.

which would look like this:

name: Rowan,
number: 0277409816

The 'name' and 'number' are the keys. They are the same across each contact in the book. The piece of data that is different is the value that each key is paired with.

In this way, when you open your contact book (if you’re 500 years old) and turn to any random page, you know that the phone number is the value paired with the key 'Phone Number'.
